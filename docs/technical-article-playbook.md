# 技術記事制作プレイブック（Zenn / 厳しめ運用版）

このプレイブックは「公開できる」ではなく「反応が取りにいける」品質を狙うための基準。
対象は自作ツール紹介記事（CLI/LSP/拡張機能）を想定。

## 1. このプレイブックの使い方

- 評価は常に **減点方式** で行う。
- まず致命的な弱点（High）を潰し、次に中位改善（Medium）を行う。
- 「情報が正しい」は前提。勝ち筋は「読者がすぐ動ける構造」。

## 2. 参考記事調査（再現手順）

### 2.1 取得

```bash
curl -s "https://zenn.dev/api/articles/<slug>"
```

### 2.2 抽出項目

- `body_letters_count`（文字数）
- `toc`（見出し数/階層）
- `body_html` 中の `<img>`, `<pre>`, `<table>` 件数
- `liked_count`（参考値、評価に直結させない）

### 2.3 注意

- `liked_count` は投稿時期・拡散経路の影響が大きい。
- 少数サンプルでは相関を断定しない。

## 3. ベースライン（目安）

- 文字数 median: `4319`
- 見出し数 median: `8.5`
- 画像数 median: `1.5`
- コードブロック median: `4`

用途別の推奨:

- 初速重視: 2500〜4500字 / 見出し6〜10 / コード3〜7 / 画像0〜3
- 保存価値重視: 4500〜8000字 / 見出し8〜14 / コード6〜15 / 画像1〜6

## 4. 厳しめレビュー基準（必須）

### 4.1 High（公開前に必ず潰す）

1. **冒頭順序が悪い**
- NG: リンク集や補足情報が最上部
- OK: `問題提起 or デモ -> 要点 -> 本文`

2. **見出しの期待値と実体験がズレる**
- NG: `30秒で試す` なのに手順が多く体感1〜2分以上
- OK: 見出しと実行コストが一致

3. **自分事化できる具体性がない**
- NG: 一般論だけ（「よくある」「〜しがち」）
- OK: `いつ/どこで/何が壊れたか` を短く明記

4. **視覚証拠がない（CLI記事）**
- NG: 画像0枚
- OK: 最低2枚（ローカル実行結果 / CI結果）

5. **CTAが抽象的**
- NG: 「試してみてください」だけ
- OK: 「最初の1アクション」を1つ明示

### 4.2 Medium（公開後の伸びを左右）

1. 見出しが多すぎて長文感が強い
2. コードブロックの意図説明がなく読み飛ばしに弱い
3. 同じ趣旨の節が離れて重複している
4. 終盤が「機能一覧」で終わり、印象が弱い

### 4.3 Low（仕上げ）

1. 日本語と英単語のスペース
2. タイトルの読みやすさ
3. topics の最適化

## 5. 構成テンプレ（自作ツール紹介）

以下の順番を固定する。

1. フック（3〜5行）
2. 30秒デモ（コマンド + 出力）
3. この記事のポイント（3〜5項目）
4. なぜ作ったか（実体験2件まで）
5. できること（3〜5項目）
6. 導入方法（最小）
7. CI連携（実運用）
8. 制限・非対応
9. まず触るリンク（GitHub / npm / 拡張）
10. CTA（最初の1アクション）

## 6. 視覚資料ルール

### 6.1 CLI/静的解析記事

最低2枚を必須とする。

- 1枚目: ローカル実行でエラーを早期検知している画面
- 2枚目: PR/Code Scanningで指摘が出る画面

任意1枚:

- 全体像図（schema -> analyzer -> diagnostics）

### 6.2 VS Code拡張記事

- 最低3枚（一覧、編集、結果）
- 推奨5枚（+検索、操作導線）

## 7. 文体ルール（非対立 + 記憶に残す）

- 他ツールを下げない
- 過度な断定をしない
- 1段落1メッセージ
- 1文60〜80文字程度
- 抽象語を減らし、具体語で書く
- 比較を書く場合は「棲み分け」を示す
- 本文でコロン（`:`）を多用しない。見出し直下や本文ラベルでは使わない

## 8. 採点フォーマット（毎回同じ型で出す）

レビュー結果は以下の順序で出力する。

1. **Findings（重大度順）**
2. **Evidence（該当行/該当節）**
3. **Fix案（最小変更で）**
4. **30分でやること / 半日でやること**

採点ランク:

- A: 公開推奨（High 0件）
- B: 公開可（High 1件まで）
- C: 要修正（High 2件以上）

## 9. 公開前ゲート

以下を満たさない限り公開しない。

- [ ] Highが0件
- [ ] コマンドを上から順に実行して再現確認済み
- [ ] YAMLを単体コピペして動作確認済み
- [ ] 視覚資料ルールを満たす
- [ ] CTAが1アクションで明確

## 10. sqlsift記事に当てる時の重点

- 冒頭を `デモ先行` にする（リンク節は後ろ）
- `30秒` 見出しと実作業量の一致を取る
- 画像2枚を追加する
- 末尾CTAを「npx実行して結果共有」など1アクションに絞る
